# Discrepancies found between the published wiki and the actual code

**This file is a change log, not a wiki page.** Do not publish it to the GitHub wiki. It records every claim the old wiki got wrong, so you can sanity-check the rewrites and — if you want — cross-reference each fix against the code yourself before committing.

Every entry follows the same shape: **what the old wiki said → what the code actually does → where I verified it**. All file paths are relative to the repo root (`/sessions/friendly-determined-dirac/mnt/chub/` in this environment; `chodeus/chub` on GitHub).

## Cancellation status of modules

**Old wiki (and this session's first-pass draft of Troubleshooting.md):** listed only five modules as cancellable (`upgradinatorr`, `jduparr`, `nohl`, `sync_gdrive`, `unmatched_assets`) and seven as non-cancellable (`poster_renamerr`, `labelarr`, `nestarr`, `renameinatorr`, `health_checkarr`, `border_replacerr`, `poster_cleanarr`).

**Actually:** 11 of the 12 modules call `is_cancelled()`. The only one that does not is `border_replacerr`.

**Verified by:**
```
grep -rl 'is_cancelled()' backend/modules/
```
Returns 11 files — everything except `border_replacerr.py`.

**Fixed in:** `Modules.md`, `Troubleshooting.md`, `FAQ.md`.

## `poster_cleanarr` modes

**Old wiki:** implied two modes (`report` for dry-run, `clean` to delete). The first-pass draft of `Troubleshooting.md` carried this forward with "Switch to `mode: clean` to actually delete."

**Actually:** six modes — `report`, `move`, `remove`, `restore`, `clear`, `nothing`. There is no `clean` mode.

**Verified by:** `backend/modules/poster_cleanarr.py`, line 27:
```python
VALID_MODES = {"report", "move", "remove", "restore", "clear", "nothing"}
```

**Fixed in:** `Modules.md` (poster_cleanarr section), `Troubleshooting.md` (poster_cleanarr does nothing section).

## Media API endpoint paths

**Old wiki:** `GET /api/media`, `PUT /api/media/{id}`.

**Actually:** `GET /api/media/search` is the list endpoint; `PUT /api/media/{media_id}/metadata` is the edit endpoint. There is no bare `GET /api/media` or `PUT /api/media/{id}`.

**Verified by:** `backend/api/media_api.py`:
- `router = APIRouter(prefix="/api/media", ...)`
- `@router.get("/search", ...)`
- `@router.put("/{media_id}/metadata", ...)`

**Fixed in:** `API.md`, `UI-Guide.md` (media/manage editing section).

## Posters API endpoint paths

**Old wiki:** `GET /api/posters` as the list endpoint.

**Actually:** there is no bare `GET /api/posters`. The list endpoint is `GET /api/posters/list`. Poster-specific routes like `/{poster_id}`, `/{poster_id}/thumbnail`, `/{poster_id}/download` all exist and work as documented.

**Verified by:** `backend/api/posters.py`:
- `router = APIRouter(prefix="/api/posters", ...)`
- `@router.get("/list", ...)`

**Fixed in:** `API.md`.

## Instance sub-routes (`quality-profiles`, `root-folders`, `tags`)

**Old wiki:** `GET /api/instances/{type}/{name}/quality-profiles`, `GET /api/instances/{type}/{name}/root-folders`, `GET /api/instances/{type}/{name}/tags`, `POST /api/instances/{type}/{name}/test`, `GET /api/instances/{type}/{name}/libraries`.

**Actually:** none of the `quality-profiles`, `root-folders`, `tags` endpoints exist. Instances use a different URL shape — `{instance_id}` instead of `{type}/{name}`:
- `POST /api/instances/test` (body describes candidate config)
- `POST /api/instances/{instance_id}/test`
- `GET /api/plex/{instance}/libraries` (Plex libraries endpoint is not under `/api/instances/`)

**Verified by:** `backend/api/instances.py`. Every route there takes `{instance_id}` or the numeric instance id; Plex libraries have their own `/plex/{instance}/libraries` path.

**Fixed in:** `API.md` (Instances section rewritten), `UI-Guide.md` (Settings → Instances section simplified).

**Verified by:** `backend/api/schedule.py`.

**Fixed in:** `API.md` (removed the `/next-runs` entry).

## System endpoints prefix

**Old wiki:** put everything under `/api/system/*` — `GET /api/system/version`, `POST /api/system/backup`, `GET /api/system/backups`, `POST /api/system/restore`, `GET /api/health` as a separate unauthenticated probe.

**Actually:** the `system.py` router has `prefix="/api"` (not `/api/system`). Most routes live at the top-level of `/api`:
- `GET /api/version`
- `GET /api/health`
- `GET /api/directory`
- `POST /api/folder`
- `POST /api/test`
- `POST /api/backup`
- `GET /api/backups`
- `POST /api/restore`

Only three routes within `system.py` keep the `/system/` prefix:
- `GET /api/system/health/snapshots`
- `GET /api/system/digest`
- `GET /api/system/cleanup-candidates`

**Verified by:** `backend/api/system.py` lines 27–29 and all `@router.get/post` paths below.

**Fixed in:** `API.md`, `Troubleshooting.md` (uses `GET /api/version` instead of `/api/system/version`), `FAQ.md` (`POST /api/backup` instead of `/api/system/backup`).

## Webhook endpoints

**Old wiki:** listed endpoints but was vague about authentication semantics and which event types trigger which actions.

**Actually:** five endpoints exist (matches the wiki list), optional shared-secret auth via `X-Webhook-Secret` header or `?secret=` query param, test events detected by substring match on `eventType`, duplicate events debounced within 5 seconds by a SHA-256 hash of `(title, year, tmdb_id, tvdb_id, imdb_id, event_type)`.

**Verified by:** `backend/api/webhooks.py`.

**Fixed in:** `Webhooks.md` (added the debounce note, clarified test-event behaviour, left the endpoint list as-is).

## `--reset-auth` CLI flag

**Old wiki:** no mention of any command-line auth reset; the only documented reset was editing `config.yml` manually.

**Actually:** `main.py` implements a `--reset-auth` argparse flag that clears username, password_hash, and jwt_secret in one go, then exits. Run via `docker compose run --rm chub python3 main.py --reset-auth`.

**Verified by:** `main.py` lines 331–335 (argparse definition) and 339–356 (`reset_auth()` implementation), 362–363 (main() dispatch).

**Fixed in:** `Installation.md` (Resetting the admin password section), `FAQ.md` (Forgot admin password), `Troubleshooting.md` (Forgot admin password).

## Default PUID / PGID

**Old wiki:** often implied `1000:1000` as defaults.

**Actually:** the container's entrypoint defaults to `PUID=100` / `PGID=99` (Unraid-style). Most Linux users still want `1000:1000` — but you have to set them explicitly.

**Verified by:** `scripts/start.sh` lines 6–7:
```bash
PUID=${PUID:-100}
PGID=${PGID:-99}
```

**Fixed in:** `Installation.md` (recommended values called out per OS), `FAQ.md` (container user/permissions section).

## Login rate limiter

**Old wiki:** didn't document it.

**Actually:** token bucket at `rate=0.2, burst=5` — roughly one attempt per 5 seconds with a burst of 5. Exceeded requests return `429`.

**Verified by:** `backend/util/rate_limiter.py` line 88:
```python
login_limiter = RateLimiter(rate=0.2, burst=5)  # 1 per 5s, 5 burst
```

**Fixed in:** `API.md` (auth section), `Troubleshooting.md` (login returns 429 section).

## Poster cache backfill

**Old wiki:** didn't explain why the low-resolution filter could return an empty list.

**Actually:** the `width` / `height` columns on `poster_cache` are populated lazily. Until you run `POST /api/posters/backfill-dimensions`, older rows have null dimensions and therefore can't be filtered by resolution.

**Verified by:** `backend/api/posters.py` backfill endpoint around line 1916.

**Fixed in:** `Troubleshooting.md` (low-resolution filter section), `UI-Guide.md` (poster stats page).

## SSE event endpoint and auth fallback

**Old wiki:** didn't explain that `EventSource` can't send custom headers.

**Actually:** `GET /api/modules/events` accepts a JWT in `?token=<jwt>` precisely because browsers don't allow setting `Authorization` on an `EventSource`. The backend accepts both the query param and the header for this endpoint.

**Verified by:** `backend/api/modules.py` events route definition.

**Fixed in:** `API.md` (modules section), `Troubleshooting.md` (real-time updates section).

## `border_replacerr` holiday schedule format

**Old wiki:** didn't document the format.

**Actually:** each holiday's `schedule` is `MM-DD:MM-DD` (year-agnostic). If two holidays' ranges overlap, the one listed first in the `holidays` array wins.

## Path-safety / SSRF guards

**Old wiki:** didn't mention either guard.

**Actually:**
- SSRF guard blocks outbound calls to reserved, cloud-metadata, and link-local ranges (so a mis-set instance URL can't probe internal endpoints).
- Path-safety guard rejects `hash_database`, `sync_location`, `gdrive_sa_location`, and folder IDs containing null bytes or starting with `-`.

**Fixed in:** `Configuration.md` (About URLs callout), `Modules.md` (jduparr + sync_gdrive gotchas), `Troubleshooting.md` (sync_gdrive invalid-path section).

## Review workflow I'd recommend before committing

1. Read each rewritten page against its predecessor.
2. Spot-check the endpoints called out in this file by opening `/docs` (FastAPI's generated Swagger UI) on a running CHUB instance and confirming the live routes match.
3. Run any module in `dry_run: true` first if you're anxious about the "Gotcha" bullets — every claim about runtime behaviour is from reading code, not running it.
4. When you're happy, move the `.md` files to `wiki/` (or wherever your wiki submodule lives) and commit. Leave **this** file out of the commit — or move it to a `docs/internal/` folder as a project change-log.
